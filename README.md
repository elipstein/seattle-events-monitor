# Seattle Events Monitor

A personal "departures board" for things worth showing up for — author talks,
chef dinners, and one-off happenings in Seattle, plus genuinely big-deal
events within a few hours' flight (Jose Andrés doing a dinner in SF, a
must-see author swinging through Portland, etc.).

Not a general event aggregator. The goal is signal, not volume: don't miss
things in your own backyard, and don't miss the rare big thing nearby either.

## The two lanes

The board has two lanes because "broad but local" and "big but far" need
different collection strategies.

### 1. In Your Backyard — Seattle, broad net

Everything from a curated list of Seattle venues/sources that reliably host
authors, chefs, and interesting speakers: bookstores, Town Hall, museums,
food/wine orgs, etc. Because the geographic radius is small, it's fine to
pull *everything* from these sources — a low-key author reading at Third
Place Books is exactly the kind of thing this lane exists to catch.

### 2. Worth the Flight — nearby metros, narrow net

Casting a wide net across Portland, the Bay Area, LA, Vancouver BC, Denver,
etc. would produce thousands of irrelevant local shows. Instead this lane is
precision-first:

- **Watchlist search**: a curated list of people/acts you actually care about
  (`config/watchlist.yml`) — chefs, authors, comedians, musicians, speakers.
  Each nearby metro gets searched by name via the Ticketmaster/SeatGeek APIs.
  This is how a Jose Andrés dinner in SF gets caught even if you've never
  heard of the specific pop-up.
- **Big-venue sweep**: independent of the watchlist, a secondary sweep pulls
  events at large venues (arenas, amphitheaters, major theaters — anything
  above a configurable capacity threshold) in those same metros, so a
  genuinely huge touring act you *hadn't* thought to watchlist still surfaces.

"A few hours" is defined in `config/regions.yml` as flight time from SEA,
default cutoff 3.5 hours. Easy to widen/narrow per-region.

## Data sources

| Source | Used for | Notes |
|---|---|---|
| [Ticketmaster Discovery API](https://developer.ticketmaster.com/products-and-docs/apis/discovery-api/v2/) | Both lanes | Free tier: 5,000 calls/day, 5 req/s. Supports keyword search (watchlist) and venue/classification filters (big-venue sweep). Primary source. |
| [SeatGeek API](https://platform.seatgeek.com/) | Big-lane, secondary | Free client id, no secret needed for search. Cross-checks Ticketmaster for coverage gaps (comedy, some theater). |
| Seattle venue websites (Elliott Bay Books, Third Place Books, Town Hall Seattle, etc.) | Local lane | No public RSS/ICS as of writing — see `scripts/fetch_local_html.py`. Lightweight scraping with per-site selectors, degrades gracefully if a site's markup changes. |
| `config/watchlist.yml` | Big lane | Manually curated. This list *is* the product — spend time on it, not on the scrapers. |

Explicitly **not** used: Eventbrite's public search API (killed in Feb 2020 —
only self-owned-event endpoints remain) and Bandsintown's API (artist-scoped,
requires per-artist written permission from Bandsintown — not usable for
arbitrary lookups by a third party).

## Repo layout

```
config/
  regions.yml        # nearby metros + flight hours from SEA + big-venue capacity threshold
  watchlist.yml       # curated names to watch for in nearby metros
  local_sources.yml   # Seattle venues/sources for the broad local lane
scripts/
  fetch_ticketmaster.py
  fetch_seatgeek.py
  fetch_local_html.py
  merge_and_classify.py
  build_ics.py
  run_all.py           # orchestrator, entry point for the GH Action
  requirements.txt
site/
  index.html          # the actual monitor/board UI
  style.css
  app.js
data/
  events.sample.json  # seed data so the board isn't empty before first run
  events.json          # generated by the daily job (committed after each run)
  events.ics            # generated calendar feed, subscribable from Google/Apple Calendar
```

The GitHub Actions workflow lives at the repo root
(`.github/workflows/seattle-events-update.yml`) because Actions only
discovers workflows there, not in subfolders.

### Adding a manual event

For sources that can't be scraped (chef dinners on Tock/Resy, mostly - see
`config/local_sources.yml`), add entries by hand to `data/manual_events.json`.
`merge_and_classify.py` picks them up on the next run same as any other
source:

```json
[
  {
    "id": "manual:canlis-jose-andres-2026-09-12",
    "title": "José Andrés x Canlis",
    "url": "https://canlis.com/",
    "venue": "Canlis",
    "city": "Seattle",
    "state": "WA",
    "start": "2026-09-12T18:00:00-07:00",
    "lane": "local",
    "category": "chef_dinners",
    "source": "manual",
    "matched_watchlist_name": "José Andrés",
    "flight_hours": null
  }
]
```

## How it runs

1. A daily scheduled GitHub Action runs `scripts/run_all.py`.
2. It fetches from Ticketmaster + SeatGeek (both lanes) and does a best-effort
   scrape of local venue pages (local lane only).
3. `merge_and_classify.py` dedupes, tags each event with `lane: local` or
   `lane: big`, computes `flight_hours` for big-lane events, and writes
   `data/events.json`.
4. `build_ics.py` regenerates `data/events.ics` from the same data, so the
   board can also be subscribed to as a calendar.
5. The workflow commits the updated `data/` files back to the branch.
6. `site/index.html` is a static page that fetches `data/events.json`
   client-side — no backend, works on GitHub Pages as-is.

## Setup

1. Get a free Ticketmaster API key: https://developer.ticketmaster.com/
2. Get a free SeatGeek client id: https://platform.seatgeek.com/
3. Add both as repository secrets: `TICKETMASTER_API_KEY`, `SEATGEEK_CLIENT_ID`
4. Edit `config/watchlist.yml` with the people/acts you actually care about.
5. Edit `config/regions.yml` if you want to widen/narrow the "few hours"
   radius or adjust which metros are included.
6. Push to `main` — the workflow runs on its own schedule from there, or
   trigger it manually via the Actions tab (`workflow_dispatch`).

To run locally:

```bash
cd scripts
pip install -r requirements.txt
export TICKETMASTER_API_KEY=...
export SEATGEEK_CLIENT_ID=...
python run_all.py
python -m http.server 8000 --directory ../site   # then open localhost:8000
```

## Roadmap

- **v1 (this)**: static board + daily data refresh + ICS subscription feed.
- **v2**: local-venue scraping hardened with a headless browser fallback for
  JS-rendered calendars (Tock/Resy-hosted chef dinners are the hard case —
  most don't expose stable HTML, may end up curated by hand in
  `config/local_sources.yml` instead of scraped).
  ICS is a fine v1 for "notify me" — a scheduled diff-and-email step could
  send a digest, but that's out of scope until the data pipeline is
  trustworthy.
- **v3**: a small ranking score per event (distance, watchlist match
  strength, days-until) so the board can sort by "most worth planning around"
  instead of pure chronology.
